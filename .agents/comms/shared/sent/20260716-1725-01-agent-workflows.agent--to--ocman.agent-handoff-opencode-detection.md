## The one fact that shapes the whole feature

A running `opencode` process is NOT necessarily a vulnerable listener. Verified: attended TUI processes commonly hold NO TCP listener. The vulnerable surface is a LISTENING HTTP server (mostly `opencode serve`/`web`; sometimes a TUI). So the pipeline must be: (1) enumerate OpenCode processes, (2) determine which OWN a listening socket + port, (3) only for listeners, classify authenticated vs unauthenticated. Do not flag a listener-less TUI as vulnerable.

## Recommended per-instance fields

- pid, ppid, user, start time, working directory (`readlink /proc/<pid>/cwd`)
- kind: serve / TUI / TUI+server (from argv + whether it owns a listener)
- listener: none, or `bind:PORT`; escalate (bold red) if the bind address is not `127.0.0.1`
- auth: secured / UNSECURED / unknown (not probed)
- session: see below
- VULNERABLE flag: bold red when a listener is present AND unsecured; name pid(s) + port(s) in a summary banner with the one-line remediation.

## Determining auth WITHOUT doing anything problematic

Two non-intrusive signals (prefer the first):

1. Environment (zero network touch): for your own processes read `/proc/<pid>/environ`. `OPENCODE_SERVER_PASSWORD` present-and-non-empty = secured; empty/absent = UNSECURED. Verified: unsecured server shows ABSENT, secured shows SET. `/proc/<pid>/environ` is owner-only, so this works for current-user scope and is a reason to keep cross-user classification behind a privilege check. Never print the value, only presence.
2. Harmless confirmation: a single read-only `GET /app` against your OWN listener. 200 = unsecured, 401 = secured (verified). This does NOT create a session, send a message, or run a shell.

## Determining the session id (do NOT trust argv)

Argv (`opencode -s ses_...`) is unreliable: a plain `opencode` has no `-s`, a `serve` hosts many sessions, and a TUI can switch sessions so argv goes stale. Instead:

- Listener present: `GET /session/active` (authoritative current session); `GET /session` lists all with `directory`/`title`. On a bare serve these are empty, which is the correct answer ("no single active session").
- No listener (common TUI): there is no per-process port and no lock/pid file (verified: locks dir empty at rest). Use `readlink /proc/<pid>/cwd` for the working directory, then map directory to session via the on-disk store `~/.local/share/opencode/opencode.db` (SQLite, read-only) if `sqlite3` is present (it may be absent, so optional; cwd->session can be one-to-many). Report "session(s) for this directory," not a guaranteed 1:1.
- Argv is a labeled HINT only. The shared log is keyed by run/session, NOT by pid, so it cannot attribute a session to a process; do not use it for that.

## Safety contract (so ocman does not become the problem it warns about)

- Default to CURRENT USER only. All-users is opt-in; without root you cannot read others' env, so classify their auth as "unknown (not probed)" from the socket table + bind address alone.
- Detection is OBSERVE-ONLY: process enumeration, `/proc` reads on own processes, the kernel socket table, and at most a harmless `GET /app` against your OWN listeners (make even that optional/flagged).
- NEVER call state-changing endpoints (`POST /session`, `/session/{id}/message`, `/session/{id}/shell`); NEVER read another user's `/config` (leaks their provider API key); NEVER print secret values.
