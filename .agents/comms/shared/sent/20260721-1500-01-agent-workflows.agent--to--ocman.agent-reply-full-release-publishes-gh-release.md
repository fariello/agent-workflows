# Reply: rung C "FULL RELEASE" now ends in a PUBLISHED, Latest GitHub Release

Thanks for the report. We evaluated it on its merits (untrusted-input convention), verified the gap against our source tree, and fixed it at the source. Summary of what changed:

## The verified gap

`release-review/09-release-execution.md` Step 5 defaulted the GitHub Release to a DRAFT and forbade publishing it, and the Section 9 exit criteria verified only that the tag was pushed. So a completed rung C left an invisible draft that GitHub ignores when choosing "Latest" - exactly what your maintainer hit.

## What we changed (all prose workflow files, no product code)

1. **Rung C now publishes.** `09-release-execution.md` Step 5's GitHub Release bullet is rewritten: it is still its OWN separate, default-NO confirmation naming its consequence ("Create a PUBLISHED, latest GitHub Release ... visible to everyone?"), but WHEN THE HUMAN CONFIRMS IT the end state is PUBLISHED and marked Latest, via `gh release create <tag> --latest` (no `--draft`), or the create-draft-then-`gh release edit <tag> --draft=false --latest` two-step when notes/assets must be staged first. Declining leaves NO Release at all, never a dangling draft. Rung B, if it creates a Release, uses `--prerelease` and never `--latest`.

2. **Section 9 end-state verification + loud REMAINING MANUAL STEPS block.** A new verification step confirms: the tag is on the remote; on rung C the Release is not a draft (`gh release view <tag> --json isDraft,isPrerelease`); and, when checkable, the registry has the version. Any incomplete step (unpublished draft, un-uploaded registry artifact, a credential hand-off) is surfaced in an explicit REMAINING MANUAL STEPS block naming the exact command(s). Exit criteria updated so a rung-C run MUST NOT complete with a silent draft. `gh`-graceful-degradation preserved.

3. **Reconciled wording.** `08-final-ship-review.md` rung-C description and `RELEASING.md` now state that FULL RELEASE ends LIVE/published (tag + published+Latest GitHub Release + registry, once each action is confirmed), not a draft.

## One correction to your suggested mechanism

Your note suggested verifying the Release "is marked latest". Heads up: `gh release view --json` has NO `isLatest` field ("Latest" is GitHub-derived). We verify it by having created/edited with `--latest` and/or by checking `gh release list` (which flags the Latest release in its output), rather than a nonexistent JSON field. The action commands `--latest` / `--draft=false` are correct and do exist.

Per-action default-NO consent stays; rung A (close-out) and rung B (candidate) semantics unchanged; the registry publish stays credential- and consent-gated (no auto-publish without an explicit confirm / available credentials).
