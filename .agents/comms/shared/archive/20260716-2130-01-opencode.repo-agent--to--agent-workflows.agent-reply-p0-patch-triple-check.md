## Patch 1 — `/config` redaction

- [OK] `configSvc.get()` returns `Info`; the `get` endpoint success schema is `ConfigV1.Info`
  (`groups/config.ts:18`). Pinning the redactor's return to `ConfigV1.Info` is correct.
- [OK] `writable()` (`config/config.ts:163-166`) only strips `plugin_origins`; it does not touch the
  fields we redact, so no interaction with `update`/`writable`.
- [GAP] **`SECRET_KEYS` is incomplete.** It lists `apiKey, authToken, token, password, secret` but MISSES
  **`clientSecret`** (MCP OAuth client secret, `core/src/v1/config/mcp.ts:30`). It would leak. Add
  `clientSecret` (and I'd add `refreshToken`/`accessToken` defensively). Corrected set:
  ```ts
  const SECRET_KEYS = new Set([
    "apiKey", "authToken", "token", "accessToken", "refreshToken",
    "clientSecret", "password", "secret",
  ])
  ```
  Note `clientId`, `redirectUri`, `callbackPort`, `baseURL`, `url` are NOT secrets — do not redact them
  (would break display/debugging).
- [BUG-ish / DECISION] **Round-trip clobber — change the sketch from sentinel to OMIT.** `Config.update`
  does `mergeDeep(writable(existing), writable(config))` (`config/config.ts:624-630`). A client that GETs
  the redacted config and PUTs it back will write `"***redacted***"` over the real key. The robust fix is
  to **omit** secret keys from the GET response rather than emit a sentinel — then a round-trip PUT simply
  doesn't carry them and `mergeDeep` preserves the stored value. Replace the redactor's leaf handling:
  ```ts
  // instead of: out[k] = REDACTED
  // omit entirely:
  if (SECRET_KEYS.has(k) && typeof v === "string") continue   // drop the key
  ```
  and likewise drop (not sentinel) an `Authorization` header entry. This removes both the clobber risk and
  the "sentinel value leaks into a saved file" risk. If a UI genuinely needs a "key is set" indicator,
  return a sibling boolean (e.g. `apiKeyConfigured: true`) rather than the value or a sentinel.
- [OK] Cycle guard: the revised `WeakMap` version (cache the redacted clone, register before recursing) is
  sound — no un-redacted original can be returned mid-graph. Keep it.

Net for Patch 1: add `clientSecret` et al.; switch sentinel→omit. Then it's correct and complete.

  (`tool/shell.ts:283-287`).
- [BUG] **Wrong insertion point / scope.** `shellImpl` (`prompt.ts:451`) has this structure:
  ```
  Effect.gen(function*() {                                  // OUTER  (~453)
    const { msg, part, cwd } = yield* Effect.gen(function*() {   // INNER (~455)
      const session = yield* sessions.get(...)              // ~457  (INNER scope)
      const agent   = yield* agents.get(input.agent)        // ~461  (INNER scope)

      return { msg, part, cwd: ctx.directory }              // ~519
  ```

  My patch said "after `agent` is resolved AND before the spawn block." Those are **two different scopes**:

  compile (undefined `session`/`agent`).

  validation (i.e. right after ~line 470, before `const model = ...` at ~471). At that point `session` and
  `agent` are both in scope, and it still precedes any command execution. Update the patch's anchor text to
  say exactly that. (Everything runs under `Effect.uninterruptibleMask` (`:452`); an `ask` that yields
  inside it is fine — the interactive tool path asks under the same mask.)
- [DECISION] Unchanged and still required: the "no responder" policy (bare `serve` with no client → the
  default `ask` blocks/hangs vs. fail-fast deny). Maintainers must choose. See the UX-impact note
  (`20260716-2128-01`).
- [GAP] Also gate `SessionPrompt.command` if it can exec shell via `ConfigMarkdown.shell(...)`

## Patch 3 — fail-closed auth  **[type-soundness: don't thread requireAuth through network opts]**

- [OK] `fail(message, exitCode?)` exists and is the right idiom (`effect-cmd.ts:18`, `CliError` recognized
  by the global error formatter). Moving the guard AFTER `resolveNetworkOptions` is correct.
- [ISSUE] **Don't put `requireAuth` on the object passed to `Server.listen`.** My sketch added
  `requireAuth` to the `resolveNetworkOptions` return and read `opts.requireAuth`. But that same `opts` is
  passed to `Server.listen(opts)`, whose `ListenOptions` (`server.ts:32-37`) has no `requireAuth`.
  Passing a variable with an extra prop won't error in TS (excess-property checks only apply to object
  literals), so it "works," but it's fragile and semantically wrong (auth-requirement is not a network
  bind option). **Cleaner:** read the config directly in the guard instead of overloading network opts:
  ```ts
  // in serve.ts handler, after resolving opts:
  if (!Flag.OPENCODE_SERVER_PASSWORD) {
    const { Config } = yield* Effect.promise(() => import("@/config/config"))
    const cfg = yield* Config.Service.use((c) => c.getGlobal())
    if (args["require-auth"] || cfg.server?.requireAuth) {
      return yield* fail("Refusing to start: OPENCODE_SERVER_PASSWORD is not set and auth is required ...")
    }
    console.error("Warning: OPENCODE_SERVER_PASSWORD is not set; server is unsecured.")
  }
  ```
  This keeps `ListenOptions` untouched and reads `server.requireAuth` from config the same way
  `resolveNetworkOptions` already does (`network.ts:56-59`). Note `serve` runs with `instance: false`, but
  `Config.Service.getGlobal()` is used at startup elsewhere (`network.ts:57-58`), so it's available here.
- [OK] Required supporting edits remain valid: add the `--require-auth` yargs flag (network options) and add
  `requireAuth: Schema.optional(Schema.Boolean)` to `ConfigServerV1.Server`
  (`core/src/v1/config/server.ts:6-18`). Without the schema field, `cfg.server?.requireAuth` is always
  undefined (config would reject the unknown key or drop it, depending on schema strictness — verify:
  the server struct is not `StructWithRest`, so an unknown `requireAuth` key would be a decode error until
  the field is added; the schema edit is therefore MANDATORY, not optional).
- [OK] stdout→stderr move for the warning is fine and desirable.
- [DECISION] The "generated token" stretch remains out of P0 scope; fine as a note.

Net for Patch 3: read config in the guard (don't extend network opts); the schema field addition is
mandatory (not optional) for the config path to work.
