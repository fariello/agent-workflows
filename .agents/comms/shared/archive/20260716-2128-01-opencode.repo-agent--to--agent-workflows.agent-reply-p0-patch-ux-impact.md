# What legitimate functionality the P0 patches would remove or change

Companion to the P0 patch sketches. For each patch: what a real user loses or has to change, how likely
it is to bite, and how to soften it. Verified against source at `08fb47373`.

## Patch 1 — `/config` secret redaction

**What changes:** `GET /config` no longer returns `provider.*.options.apiKey`, `*.headers.authorization`,
or other secret leaves; they come back as `"***redacted***"`.

Functionality affected:

1. **A settings UI that shows the configured API key.** Any client that reads the key back to display it
   (e.g. "your key: sk-…") can no longer do so. Impact: low; most UIs show `••••` anyway. Confirmed no
   in-repo consumer reads `options.apiKey` back from the `/config` GET response (grep of `packages/tui`,
   `packages/opencode/src/cli` found none).
2. **GET→edit→PUT round-trip config editors — REAL REGRESSION RISK.** `Config.update` does
   `mergeDeep(existing, submitted)` (`config/config.ts:624-630`). If a client GETs the (now redacted)
   config, lets the user edit some unrelated field, and PUTs the whole object back, it will write the
   literal `"***redacted***"` over the real `apiKey`, corrupting it. Today (unredacted) that round-trip
   preserves the key. Mitigations: (a) have the redactor omit secret keys entirely rather than emit a
   sentinel, so a round-trip PUT simply doesn't include them and `mergeDeep` keeps the existing value; OR
   (b) make `update` treat the `"***redacted***"` sentinel as "leave unchanged." Option (a) is cleaner
   and I'd recommend switching the sketch to omit-not-sentinel. Note the in-repo `update` handler returns
   `ctx.payload` (the caller's own submission), not a GET, so the built-in flows are safe; the risk is for
   third-party/round-trip editors.
3. **Debugging/support ("what key is opencode using?").** A user or support script can no longer read the
   key via the API. Minor; the key is still in the on-disk `opencode.json` the user controls.

Net: for normal use, nothing visible is lost. The one thing to get right is round-trip editors → prefer
omit over sentinel.

## Patch 2 — `/shell` permission gate

**What changes:** `POST /session/{id}/shell` now goes through the permission policy (default effect
`ask`). This is NOT only an API-abuse path — the **TUI's own shell mode uses it**: the `!command`
composer calls `sdk.client.session.shell(...)` (`packages/tui/src/component/prompt/index.tsx:1060`). So
this affects an everyday interactive feature.

Functionality affected:

1. **`!`-shell in the attended TUI now prompts for permission** (once per command / pattern, subject to
   "always" rules), where before it ran immediately with no prompt. Impact: medium and VISIBLE. For users
   who relied on instant `!` shell, this adds an approval step. Softening: the permission system supports
   "always allow" for a pattern, and a user/agent ruleset can pre-`allow` `bash` so the prompt only
   appears once (or never, if configured). Document that `permission` config restores no-prompt behavior
   deliberately.
2. **Headless / bare `serve` shell calls may HANG instead of running.** With no interactive client
   attached to answer the `ask`, the default effect blocks on a deferred (`core/src/permission.ts:197-217`)
   — so a scripted `POST /shell` against a bare `serve` that previously executed will now wait forever
   (until aborted). This is the deliberate "no silent exec" trade, but it is a behavior change for anyone
   using `/shell` as an unattended automation primitive. Softening / decision needed: pick a "no
   responder" policy — either (a) auto-deny API-origin shell when nothing can answer (fast failure instead
   of hang), or (b) allow an explicit `permission: allow` ruleset / a `--yolo`-style opt-in for trusted
   automation. Whichever, make "unattended `/shell`" a conscious opt-in rather than the default.
3. **Agents/automation that drove `/shell` expecting fire-and-forget** now need a responder or a
   pre-approving ruleset. Impact: medium for automation users; none for the security posture (that's the
   point).

Net: this is the most user-visible patch. It intentionally removes "shell with zero approval," including
in the attended TUI. Give users a config-level way to restore no-prompt for trusted contexts, and pick an
explicit no-responder policy so unattended callers fail fast instead of hanging.

## Patch 3 — fail-closed auth (`--require-auth` / `server.requireAuth`)

**What changes:** purely additive and OPT-IN (default `false`). Nothing is removed for existing users.

Functionality affected:

1. **Only if a user turns it on:** `serve`/`web` will refuse to start when `OPENCODE_SERVER_PASSWORD` is
   unset, instead of starting unsecured. That is the intended behavior; the "loss" is "starting an
   unsecured server while the flag is set," which is what they asked to forbid.
2. **The warning moves from stdout to stderr.** A script that scraped the unsecured-warning line from
   stdout would no longer see it there. Extremely niche; the startup "listening on …" line stays on
   stdout.

Net: no impact unless opted in; then the only lost ability is "run unsecured while requiring auth," which
is the goal.

## Related hardening (timing-safe compare) — no functional impact

Constant-time password compare changes nothing a user can observe (same accept/reject behavior).

The only patch that changes an everyday, attended-user experience is **Patch 2** (shell approval),
because the TUI's `!` shell rides the same endpoint. Everything else is invisible to normal use or opt-in.
Recommend shipping Patch 2 with a documented config path to restore no-prompt for trusted setups, and an
explicit unattended-`/shell` policy so it fails fast rather than hanging.
