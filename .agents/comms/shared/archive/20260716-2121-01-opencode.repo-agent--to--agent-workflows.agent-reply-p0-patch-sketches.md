## Patch 1 — Redact provider secrets from `GET /config` (P0, advisory 3.4)

**Problem.** `GET /config` returns `configSvc.get()` verbatim; `provider.<id>.options.apiKey` is a plain
string, so a configured key is returned in cleartext (unauth by default). `mcp.*.headers` and
`provider.*.headers` are `Record<string,string>` (`v1/config/mcp.ts:50`, `v1/config/provider.ts:62`) and
can carry `Authorization`/token secrets too.
**File.** `packages/opencode/src/server/routes/instance/httpapi/handlers/config.ts`
**Anchor.** the `get` handler (`:14-16`) currently `return yield* configSvc.get()`.
**Endpoint contract.** `get`'s success schema is `ConfigV1.Info` (`groups/config.ts:18`). The redactor
must return the SAME shape/type (`Info`) — replace secret string leaves in place, don't change structure.
**Fix.**

```diff
--- a/packages/opencode/src/server/routes/instance/httpapi/handlers/config.ts
+++ b/packages/opencode/src/server/routes/instance/httpapi/handlers/config.ts
@@
     const get = Effect.fn("ConfigHttpApi.get")(function* () {
-      return yield* configSvc.get()
+      return redactSecrets(yield* configSvc.get())
     })
```

Module-local helper (keep the return type as the config `Info`, structural + defensive):

```ts
import type { ConfigV1 } from "@opencode-ai/core/v1/config/config"

const SECRET_KEYS = new Set(["apiKey", "authToken", "token", "password", "secret"])
const REDACTED = "***redacted***"

// Deep-clone + redact secret-looking string leaves anywhere under the config,
// plus any `Authorization` header. Returns the same Info shape. Does NOT mutate
// the live config. Cycle-safe: on revisit, redact into a fresh node (never return
// an un-redacted original mid-graph).
function redactSecrets(config: ConfigV1.Info): ConfigV1.Info {
  const seen = new WeakMap<object, unknown>()
  const walk = (value: unknown): unknown => {
    if (Array.isArray(value)) return value.map(walk)
    if (value && typeof value === "object") {
      const hit = seen.get(value as object)
      if (hit) return hit
      const out: Record<string, unknown> = {}
      seen.set(value as object, out) // register BEFORE recursing so cycles resolve to the redacted node
      for (const [k, v] of Object.entries(value as Record<string, unknown>)) {
        if (SECRET_KEYS.has(k) && typeof v === "string") out[k] = REDACTED
        else if (k.toLowerCase() === "headers" && v && typeof v === "object" && !Array.isArray(v)) {
          out[k] = Object.fromEntries(
            Object.entries(v as Record<string, unknown>).map(([hk, hv]) =>
              hk.toLowerCase() === "authorization" ? [hk, REDACTED] : [hk, walk(hv)],
            ),
          )
        } else out[k] = walk(v)
      }
      return out
    }
    return value
  }
  return walk(config) as ConfigV1.Info
}
```

**Notes.**
- Scope: only config-file keys. Keys in `auth.json`/env don't flow through `/config`.
- Covers `provider.*.options.apiKey`, `provider.*.headers.authorization`, and `mcp.*.headers.authorization`
  by structural match (any depth). If maintainers prefer schema-driven redaction, annotate the secret
  fields (`v1/config/provider.ts:87`, `v1/config/mcp.ts:50`) and redact by annotation instead.
- `GET /config/providers` already routes through `Provider.toPublicInfo` (`config.ts:24-29`) and does not
  emit raw keys — leave as-is.
- **Fixed vs first draft:** cycle guard now caches the redacted clone (a `WeakMap`), so a cyclic graph can
  no longer return an un-redacted original; return type pinned to `ConfigV1.Info`.

## Patch 2 — Gate `POST /session/{id}/shell` behind the permission policy (P0, advisory 3.5)

**Problem.** The V1 `SessionPrompt.shell` path spawns with NO permission check. `shellImpl`
(`packages/opencode/src/session/prompt.ts:451-570`) builds records then `ChildProcess.make(sh, args,
{ cwd })` → `spawner.spawn(cmd)`, no `ask`. The agent *message* path DOES ask (`prompt.ts:342-349`).
**File.** `packages/opencode/src/session/prompt.ts`
**Anchor.** inside `shellImpl` (`:451`), AFTER `const agent = yield* agents.get(input.agent)` is
resolved and validated, and BEFORE the spawn block (~`:522`). The V1 permission service is in scope:
`const permission = yield* Permission.Service` (`:125`).
**Correct request shape** — this was the load-bearing bug in my first draft. `permission.ask` takes
`PermissionV1.AskInput = { ...Request.fields, id?, ruleset }`
(`packages/schema/src/v1/permission.ts:51`), where `Request.fields` (`:27-35`) requires:
`permission: string`, `patterns: string[]`, `metadata: Record<string,unknown>`, `always: string[]`,
`sessionID`, and optional `tool`. The bash tool calls it as
`{ permission: ShellID.ToolID /* "bash" */, patterns: [...], always: [...], metadata: {...} }`
(`packages/opencode/src/tool/shell.ts:283-287`; `ShellID.ToolID === "bash"` at `tool/shell/id.ts:16`).

```diff
--- a/packages/opencode/src/session/prompt.ts
+++ b/packages/opencode/src/session/prompt.ts
@@ shellImpl(...), after `const agent = yield* agents.get(input.agent)` is resolved/validated
+            // Gate API-driven shell the same way the agent tool path is gated:
+            // default effect is `ask` (blocks until answered); an explicit deny refuses.
+            yield* permission
+              .ask({
+                permission: ShellID.ToolID,            // "bash" — matches the bash tool's action id
+                patterns: [input.command],
+                always: [input.command],
+                metadata: { command: input.command },
+                sessionID: input.sessionID,
+                ruleset: Permission.merge(agent.permission, session.permission ?? []),
+              })
+              .pipe(Effect.orDie)
```

(`ShellID` is already imported in `prompt.ts` — it references `ShellID.ToolID` for the tool part at
`:514`. `Permission.merge` is already used at `:346`. So no new imports needed. Verify `session` is the
resolved session var in that scope — it is fetched near the top of `shellImpl`.)

**Semantics to decide (real design choice, not a nit):** when NO interactive client is attached (bare
`serve`), the default `ask` effect BLOCKS on a deferred — so `/shell` will HANG rather than 403. That is
strictly safer than today (no silent exec), but you may prefer explicit deny-by-default for
API-originated shell with no responder. If so, thread a `source: "api"` marker and treat an
unanswerable `ask` as deny. Flagging so the maintainers choose deliberately.

**Also:** consider the same gate on `SessionPrompt.command` if it can run shell via
`ConfigMarkdown.shell(...)` (`prompt.ts:1397-1403`).

**Fixed vs first draft:** first draft used `{ type, pattern }` — WRONG. `type`→`permission`,
`pattern`→`patterns` (array), and it was MISSING the required `metadata` and `always` fields, so it
would have failed `AskInput` schema validation at runtime. Corrected above.

## Patch 3 — Fail-closed auth option (P0, advisory 3.1 / the fail-open gotcha)

**Problem.** With no `OPENCODE_SERVER_PASSWORD`, `serve` prints a stdout warning and runs fully open
(`packages/opencode/src/cli/cmd/serve.ts:15-22`). No "refuse to start."
**Files.** `packages/opencode/src/cli/cmd/serve.ts` (guard) and, for the config key,
`packages/core/src/v1/config/server.ts` (schema) — see note.
**Idiom.** Use the project's CLI-failure helper `fail(message, exitCode?)` from
`packages/opencode/src/cli/effect-cmd.ts:18` (it fails with a `CliError` recognized by the global error
formatter). Do NOT use `Effect.fail(new Error(...))`.

```diff
--- a/packages/opencode/src/cli/cmd/serve.ts
+++ b/packages/opencode/src/cli/cmd/serve.ts
@@
-import { Effect } from "effect"
-import { effectCmd } from "../effect-cmd"
+import { Effect } from "effect"
+import { effectCmd, fail } from "../effect-cmd"
 import { withNetworkOptions, resolveNetworkOptions } from "../network"
 import { Flag } from "@opencode-ai/core/flag/flag"
@@
   handler: Effect.fn("Cli.serve")(function* (args) {
     const { Server } = yield* Effect.promise(() => import("../../server/server"))
-    if (!Flag.OPENCODE_SERVER_PASSWORD) {
-      console.log("Warning: OPENCODE_SERVER_PASSWORD is not set; server is unsecured.")
-    }
     const opts = yield* resolveNetworkOptions(args)
+    if (!Flag.OPENCODE_SERVER_PASSWORD) {
+      // opts carries the resolved config (network.ts merges config.server.*), so
+      // both the --require-auth flag and a config `server.requireAuth` are honored.
+      if (args["require-auth"] || opts.requireAuth) {
+        return yield* fail(
+          "Refusing to start: OPENCODE_SERVER_PASSWORD is not set and auth is required " +
+            "(--require-auth / server.requireAuth). Set a password or remove the requirement.",
+        )
+      }
+      console.error("Warning: OPENCODE_SERVER_PASSWORD is not set; server is unsecured.")
+    }
     const server = yield* Effect.promise(() => Server.listen(opts))
     console.log(`opencode server listening on http://${server.hostname}:${server.port}`)

     yield* Effect.never
   }),
```

Supporting edits required (my first draft omitted these):

1. **Flag** — add to the network options in `packages/opencode/src/cli/network.ts` `options`:
   ```ts
   "require-auth": {
     type: "boolean" as const,
     describe: "refuse to start unless OPENCODE_SERVER_PASSWORD is set",
     default: false,
   },
   ```
2. **Config key** — add `requireAuth` to the server schema
   `packages/core/src/v1/config/server.ts` (the `Server = Schema.Struct({...})`):
   ```ts
   requireAuth: Schema.optional(Schema.Boolean).annotate({
     description: "Refuse to start the server unless OPENCODE_SERVER_PASSWORD is set",
   }),
   ```
   and surface it from `resolveNetworkOptionsNoConfig` (`network.ts:62-79`) into the returned object
   (e.g. `requireAuth: config?.server?.requireAuth ?? args["require-auth"]`) so `opts.requireAuth` exists.
   Also add `requireAuth?: boolean` to the `ListenOptions`/returned type as needed.

**Stretch (bigger, recommended default): generated token.** When no password and `--require-auth`/service
mode, GENERATE a random token, write it `0600` to `Global.Path.state/server-auth`, and feed it into
`ServerAuth.Config` (`server/auth.ts:17-20`) so the server is never open. `ServerAuth.required()`/
`authorized()` (`:24-33`) centralize the check — that's the seam. Larger than a P0 diff; noted only.

**Related low-risk hardening (in `auth.ts`):** the password compare at `server/auth.ts:32` uses `===`
(not constant-time). For a network-exposed (`0.0.0.0`) server, use `crypto.timingSafeEqual` on
equal-length, length-guarded buffers. Trivial, expected in a security PR.

**Fixed vs first draft:** first draft referenced a `config` variable that is NOT in scope in the serve
handler (it's local to `resolveNetworkOptions`), used `Effect.fail(new Error(...))` instead of the
project's `fail(...)` helper, put the check before `resolveNetworkOptions` (so config wasn't available),
and never mentioned that `server.requireAuth` must be added to the config schema. All corrected: the
guard now runs after `resolveNetworkOptions`, reads `opts.requireAuth`, and the schema/flag edits are
listed.
