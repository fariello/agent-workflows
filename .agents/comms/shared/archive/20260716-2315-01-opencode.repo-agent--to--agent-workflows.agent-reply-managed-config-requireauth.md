# Can a managed/system config forbid unsecured servers? Source answer

Version read: OpenCode `dev` @ `08fb47373` (current on this box; behavior matches `v1.18.3`
`127bdb307`). All citations worktree-relative.

**Headline (this corrects your framing in two important, favorable-then-unfavorable ways):**

- **GOOD:** A Linux managed tier already EXISTS. `/etc/opencode/opencode.json{,c}` is read today on Linux
  (not macOS-only), and it is merged LAST among file sources, so it already WINS over user/project config
  on scalar keys. Your "macOS-only gap" premise is partly wrong — Linux parity for the *config-file*
  managed dir is already there.
- **BAD (the real gap):** The server-startup/network path does **not read the managed tier at all**, and
  CLI flags override config regardless. So today a locked `/etc/opencode` value would **not** actually
  constrain `serve`/`web` binding, and there is no `server.requireAuth` key or startup check. So this is
  **not** "reuse an existing lock" — it's "wire the existing managed dir into the startup path + add a
  key + a guard." Small-to-medium, not trivial, and not a philosophy change.

Details per question.

## Q1. Managed/admin tier reality + precedence + does it lock?

- **Load site:** `packages/opencode/src/config/managed.ts` (`systemManagedConfigDir()` :20-29,
  `managedConfigDir()` :31-33, `readManagedPreferences()` :43-69). Consumed in
  `packages/opencode/src/config/config.ts:516-534`.
- **Precedence (merge order in `loadInstanceState`, `config.ts`):** global `~/.config/opencode`
  (:398-399) → `OPENCODE_CONFIG` (:401-404) → **project `.opencode` upward from cwd** (:406-410) →
  `OPENCODE_CONFIG_DIR` dirs (:424-434) → `OPENCODE_CONFIG_CONTENT` (:468-476) → active-org console config
  (:495-505) → **managed dir `/etc/opencode` (:516-522)** → **macOS MDM plist (:524-534)**. Merge is
  `mergeConfigConcatArrays` → `mergeDeep(result, next)` (:41-46, :351-354): **later source wins on
  scalars; arrays concatenate.**
- **(a) exact site/precedence:** above. Managed dir and MDM are the LAST two file merges, so they win over
  user/project/global config for scalar keys.
- **(b) Linux or macOS-only:** **Both.** `/etc/opencode` is the `default` branch → Linux (and any
  non-darwin/non-win32) (`managed.ts:26-27`). macOS also gets `/Library/Application Support/opencode`
  (:22-23) AND the MDM `.mobileconfig` plist path (:43-56). Windows gets `%ProgramData%\opencode`
  (:24-25). So the managed **directory** tier is cross-platform TODAY; only the MDM-plist mechanism is
  macOS-specific.
- **(c) true lock or just a merge layer?** **Merge layer, not a hard lock — but it's the last file layer,
  so for CONFIG it effectively wins over user/project.** Two caveats keep it from being a real "lock":
  1. It wins by *ordering*, not by an immutable flag. Arrays (`cors`, `plugin`, `permission` lists)
     **concat** rather than being replaced, so a managed array value can't remove user-supplied entries.
  2. **CLI flags are evaluated separately and beat config** (see Q4). So managed config can't lock a value
     a user overrides on the command line.

  So: managed config wins over other *config files* for scalars, but is not authoritative over CLI flags,
  and does not "lock" arrays.

## Q2. Cross-OS parity matrix (with file:line)

|---|---|---|---|---|---|---|

| Managed dir `/Library/Application Support/opencode` | macOS | YES | Yes | No | No | `managed.ts:22-23`; `config.ts:516-522` |
| Managed dir `%ProgramData%\opencode` | Windows | YES | Yes | No | No | `managed.ts:24-25`; `config.ts:516-522` |
| MDM managed-prefs `.mobileconfig` plist | macOS only | YES | Yes (merged even after `/etc`-style dir; comment says "override everything") | No | No | `managed.ts:43-69`; `config.ts:524-534` |

**Correction to your framing:** the platform gap is NOT "macOS admins can lock, Linux can't." Linux, macOS,
and Windows all have the managed *directory* tier and it wins over user config for scalars. macOS
additionally has the MDM-plist path. The remote org-config tier is merged **early**, so it is the *weakest*
admin lever (project and managed both override it) — that's the opposite of what "org-pushed locked value"
would need.

**The real, cross-platform gap is different and more useful for your justification:** the managed tier
influences the merged *config object* but the **server bind/auth startup path never consults it, and no
`server.requireAuth`/`allowUnauthenticated` key exists.** That gap is identical on every OS. So the
compelling argument is not "close a macOS-vs-Linux hole" — it's "the managed tier that already exists on
all platforms doesn't reach server startup, and there's no auth-enforcement key for it to carry."

## Q3. Linux system path effort

**Already done** — `/etc/opencode` exists and is read (`managed.ts:26-27`, `config.ts:516-522`). You do NOT
need to add a Linux authoritative *path*. What's missing is (a) a key and (b) making the startup path read
the managed tier. See Q4.

## Q4. Enforcing requireAuth from managed config — the crux

Three concrete obstacles, all in source:

1. **No `server.requireAuth` key exists.** The server config schema is `port`/`hostname`/`mdns`/
   `mdnsDomain`/`cors` only (`packages/core/src/v1/config/server.ts:6-18`). The struct is not
   `StructWithRest`, so an unknown `requireAuth` key would be dropped/rejected on decode until the field
   is added. **Mandatory schema edit.**

2. **The startup path does not read the managed tier — it reads GLOBAL only.** `serve`/`web` resolve
   network options via `resolveNetworkOptions` (`cli/network.ts:56-60`), which calls
   **`Config.getGlobal()`**. `getGlobal` → `loadGlobal` (`config.ts:246-293`) merges ONLY
   `~/.config/opencode/{config.json,opencode.json,opencode.jsonc}` (:258-260). **Managed config
   (`/etc/opencode` + MDM) is merged in `loadInstanceState` (:516-534), NOT in `loadGlobal`.** So today an
   operator value in `/etc/opencode/opencode.json` is invisible to the network/startup path. This is the
   single most important finding: **managed config currently cannot constrain server binding at all**,
   regardless of precedence, because startup never loads it.

3. **CLI flags beat config even when config is consulted.** `resolveNetworkOptionsNoConfig`
   (`cli/network.ts:62-79`) uses `hostnameExplicitlySet ? args.hostname : config?.server?.hostname`, etc.
   So `--hostname 0.0.0.0` overrides any config hostname. A managed value could not lock the bind against
   a user's CLI flag without changing this precedence.

**Does startup have merged config in hand?** It has *global* config (`getGlobal`), not the fully-merged
instance config that includes managed/MDM. `serve`/`web` run with `instance: false`
(`cli/cmd/serve.ts:12`, `web.ts`), so there's no ambient instance context; they'd need to explicitly load
the managed tier (or a dedicated startup-policy read) themselves.

**Where the check must live to cover every listener:** a shared pre-`Server.listen` guard invoked by BOTH
`cli/cmd/serve.ts` (:13-22) and `cli/cmd/web.ts` (:40-44) — they warn-and-continue independently today —
plus any TUI/attach path that starts a listener. (This matches the "shared startup policy" conclusion from
the GPT-5.6 review.) The guard would: load global + managed config (and MDM), resolve
`server.requireAuth`/`allowUnauthenticated`/`allowNonLoopbackBind`, compare against `ServerAuth.required()`
(`server/auth.ts:24-26`) and the resolved hostname, and `fail(...)` (`cli/effect-cmd.ts:18`) before
binding.

## Q5. Philosophy / SECURITY.md conflict

**No fundamental conflict.** OpenCode *already* ships an admin-authoritative tier that overrides users:
the managed dir on all platforms (`config.ts:516-522`) and the MDM plist whose own code comment says it
**"override[s] everything"** (`config.ts:524`). So "operator can override user config" is an established,
in-code stance, not a new philosophy. Extending it to a `server.requireAuth`/bind policy is consistent with
the existing managed/MDM design. The only genuinely new concept is **letting managed config constrain CLI
flags** (a real "lock" vs. current "last-merge-wins"), which is a deliberate precedence decision maintainers
should sign off on — but even that is an extension, not a reversal, of the existing admin tier. The
SECURITY.md "users control their own config" stance already coexists with the MDM override, so this is
reconcilable.

## Q6. Effort sizing

**Small-to-medium, mostly wiring, not a new concept:**
- Add `requireAuth` (+ optionally `allowUnauthenticated`, `allowNonLoopbackBind`) to
  `core/src/v1/config/server.ts` — **small, mandatory.**
- Make the startup path load the managed tier. Two options: (a) add managed-dir + MDM merge into
  `loadGlobal` so `getGlobal()` sees it (broadest, but changes global-config semantics everywhere), or
  (b) a dedicated `resolveStartupPolicy()` that reads global + managed + MDM just for the guard
  (narrower, safer). **Medium; (b) preferred.**
- Add a shared pre-listen guard called by `serve`/`web`/TUI-listener paths. **Small-medium.**
- Decide whether managed `requireAuth`/bind policy overrides CLI flags (the only "philosophy" bit).
  **Policy decision, small code.**

Rough sizing: **~1 focused PR** (schema key + startup-policy read + shared guard + tests), plus a maintainer
policy decision on CLI-flag override. NOT a rewrite of the config system.

## Corrections to your framing (explicit, since you asked)
1. Linux managed config is **not** missing — `/etc/opencode` already works and already wins over user

2. The real gap is cross-platform and different: **managed config never reaches the server-startup/network

3. Remote org `.well-known` config is the **weakest** admin lever (merged early; project & managed override

Net: the disclosure package can accurately say "OpenCode already has a cross-platform operator config tier

# ADDENDUM (follow-up Qs): what a sysadmin can enforce via `/etc/opencode` TODAY

Two follow-up questions: (1) can a sysadmin put settings in `/etc/opencode/opencode.json` that — barring
other config/CLI overrides — (a) prevent unsecured servers and/or (b) prevent other undesirable behavior;
(2) of the settings `/etc/opencode` *does* affect, which are superseded by `~/.config/opencode` or CLI.

**The single most important thing to internalize: there are TWO config-load paths, and they differ on
whether they even read `/etc/opencode`.**

- **Instance config** = `Config.Service.get()` → `loadInstanceState` → **DOES merge `/etc/opencode` + MDM**
  (`config.ts:516-534`). Used by permissions, prompt/shell, share, etc. (`prompt.ts:124,522`;
  `permission.ts:156`).
- **Global-only config** = `Config.Service.getGlobal()` → `loadGlobal` → **does NOT merge `/etc/opencode`**;
  it reads only `~/.config/opencode/{config.json,opencode.json,opencode.jsonc}` (`config.ts:246-260`).
  **The server network/bind startup path uses THIS one** (`cli/network.ts:56-60` → `getGlobal`).

That split is what makes the answer nuanced.

## Q1(a) — Prevent unsecured servers (no password): **NO. Not possible via any config today.**

The server password is read **only** from the `OPENCODE_SERVER_PASSWORD` environment variable
(`server/auth.ts:18`: `EffectConfig.string("OPENCODE_SERVER_PASSWORD")`). **There is no `server.password`,
`requireAuth`, or `allowUnauthenticated` key in the config schema at all** (`core/src/v1/config/server.ts:6-18`
exposes only `port`/`hostname`/`mdns`/`mdnsDomain`/`cors`). Consequences for a sysadmin:

- `/etc/opencode` **cannot set** a password (no key to carry it).
- `/etc/opencode` **cannot require** a password (no fail-closed check exists; `serve.ts:15-22` and
  `web.ts:40-44` only warn to stdout, then start).
- Therefore a sysadmin **cannot** prevent a user from running an unsecured `opencode serve`/`web` through
  config. The only lever is the environment (`OPENCODE_SERVER_PASSWORD`) injected by a launcher/wrapper, or
  OS-level isolation — not `/etc/opencode`.

## Q1(b) — Prevent OTHER undesirable behavior: **PARTIALLY — and NOT the network-exposure ones.**

Split by which load path consumes the key:

**Can be influenced by `/etc/opencode` today (instance-config path, managed IS merged):**
- `permission` — tool/shell approval ruleset (`permission.ts:156` via `configured()`). A managed `deny`
  rule for, e.g., shell/edit would apply. NOTE: this only gates the AGENT tool path; it does **not** gate
  the direct `POST /session/:id/shell` endpoint, which bypasses permissions entirely (prior finding). So a
  managed `permission` deny does **not** stop the load-bearing cross-user shell primitive.
- `share: "disabled"` — blocks session sharing (`share/session.ts:28`).
- `autoupdate: false` — disables auto-update (`cli/upgrade.ts:10`).
- `tools`, `mcp`, `disabled_providers`, `instructions`, etc. — all merged via instance config.

**CANNOT be locked by `/etc/opencode` today (network/bind — the exposure-relevant ones):**
- `server.hostname`, `server.mdns`, `server.port` — resolved through `resolveNetworkOptions` →
  **`getGlobal()`**, which **excludes** `/etc/opencode` (`network.ts:56-60`, `config.ts:246-260`). So a
  managed `server.hostname: "127.0.0.1"` or `server.mdns: false` in `/etc/opencode` has **no effect** on
  what the server binds to. This is the critical gap: **the settings most relevant to your disclosure
  (loopback-only, no mDNS) are exactly the ones `/etc/opencode` cannot currently constrain.**

**So Q1(b) verdict:** a sysadmin can nudge some behavior (permissions for the agent path, disable share,
disable autoupdate) via `/etc/opencode`, but **cannot** lock the network bind address, **cannot** disable
mDNS, and **cannot** require/prevent-absence-of a password — i.e. cannot prevent the actual
unsecured/network-exposed-server concerns.

## Q2 — Of settings `/etc/opencode` DOES affect (instance-config keys), what supersedes them?

For the instance-config keys (permission, share, autoupdate, tools, mcp…), `/etc/opencode` is merged
**LAST among files** (`config.ts:516-522`, then MDM at :524-534), so:

- **`~/.config/opencode/opencode.json` does NOT override `/etc/opencode`** for scalar values — managed is
  merged *after* global/project, so **managed wins** (`mergeDeep(result, next)`, later source wins,
  `config.ts:41-46,351-354`). Same for project `.opencode`, `OPENCODE_CONFIG`, `OPENCODE_CONFIG_DIR`,
  `OPENCODE_CONFIG_CONTENT`, and org/console config — all merged BEFORE managed, so all lose to managed on
  scalars. (macOS MDM plist beats even `/etc`-style managed dir.)
- **EXCEPTION — arrays concatenate, they don't replace** (`mergeConfigConcatArrays`, `config.ts:45-46`).
  So for array-valued keys (`cors`, `plugin`, and the permission *lists*), a user/project config can **add**
  entries that a managed array cannot remove. A managed `permission` deny is not guaranteed to be the
  final say if a later-concatenated allow rule matches — verify rule precedence per your policy.
- **Command-line flags: N/A for these keys.** There are no CLI flags for `permission`/`share`/`autoupdate`,
  so CLI doesn't supersede them. CLI override matters only for the network keys — which, as noted, don't
  read managed config anyway.

**And for the network keys specifically (even though managed doesn't reach them):** if the startup path
*were* wired to read managed config, note that **explicit CLI flags still beat config**
(`network.ts:62-74`: `hostnameExplicitlySet ? args.hostname : config?.server?.hostname`). So `--hostname
0.0.0.0` / `--port` / `--mdns` on the command line would override a managed value. That's the other reason
managed config isn't a true "lock" for binding: not-consulted today, and CLI-overridable even if it were.

## One-paragraph summary for the disclosure

Today, `/etc/opencode/opencode.json` can override user and project config (scalar keys) and is thus useful
for setting some behavioral defaults that run through instance config — `permission`, `share: "disabled"`,
`autoupdate: false`. But it is **not** an effective security lock for the concerns in this advisory: it
**cannot** set or require a server password (password is env-var-only, no config key exists), it **cannot**
constrain the server bind address or disable mDNS (network binding is resolved from *global-only* config
that excludes the managed tier, and is CLI-overridable), and array-valued keys only concatenate. So a
sysadmin cannot, via config alone, prevent a user from starting an unsecured or `0.0.0.0`-bound server.
That is precisely why the proposed fix requires (1) a new `server.requireAuth`/bind-policy KEY, (2) making
the server-startup path READ the managed tier (it currently uses `getGlobal`, which skips it), and (3) a
precedence decision so a managed lock beats CLI flags. Without all three, `/etc/opencode` provides
behavioral defaults, not enforceable security controls.

# ADDENDUM 3 (follow-up): proposal — hard-require a non-empty server password before listen

Two mechanisms, as a tier: an **env var** that is *minimally acceptable* (trivial code, but a local shell
user can unset it), and a **managed config key** that is *preferred* (operator-owned in `/etc/opencode`,
user-unbypassable). Recommendation: the config key is the correct durable control; the env var is only an
interim stopgap for service/launcher deployments.

## Common seam (both options use it)

The single choke point every listener path passes through is **`Server.listen()`**
(`server/server.ts:73`, effect body `listenEffect` at `:83`). `serve` (`cli/cmd/serve.ts`), `web`
(`cli/cmd/web.ts`), and any TUI/attach fallback all call it, so a guard here covers every path and none can
be missed (the fix for the "`serve.ts`-only patch misses `web.ts`" gap). Password is read only from the
`OPENCODE_SERVER_PASSWORD` env var (`server/auth.ts:18`); "set" means non-empty, matching
`ServerAuth.required()` (`auth.ts:24-26`). The guard just enforces presence.

## Option 1 (minimally OK): `OPENCODE_REQUIRE_SERVER_PASSWORD` env var

**Robustness: LIMITED.** Only as strong as the sysadmin's control of the environment. Fine for a systemd
unit / locked launcher, but **a shell user can `unset OPENCODE_REQUIRE_SERVER_PASSWORD`** and start
unsecured. Stopgap, not enforcement on a multi-user host. **Effort: trivial** (~1 flag + ~6 lines).

Declare in the Flag module (`core/src/flag/flag.ts`, using the `truthy` helper at `:3`):

```diff
   OPENCODE_DISABLE_AUTOUPDATE: truthy("OPENCODE_DISABLE_AUTOUPDATE"),
+  OPENCODE_REQUIRE_SERVER_PASSWORD: truthy("OPENCODE_REQUIRE_SERVER_PASSWORD"),
```

Guard in `Server.listen` (`server/server.ts`, top of `listenEffect`, before `startWithPortFallback`):

```diff
 const listenEffect = Effect.fn("Server.listen")(
   function* (opts: ListenOptions) {
+    const password = process.env.OPENCODE_SERVER_PASSWORD
+    if (Flag.OPENCODE_REQUIRE_SERVER_PASSWORD && (!password || password.trim() === "")) {
+      return yield* Effect.die(
+        new Error(
+          "Refusing to start server: OPENCODE_REQUIRE_SERVER_PASSWORD is set but " +
+            "OPENCODE_SERVER_PASSWORD is empty. Set a non-empty password.",
+        ),
+      )
+    }
     const state = yield* startWithPortFallback(opts)
```

(Requires importing `Flag` in `server.ts`.) A sysadmin sets it via systemd `Environment=`, a forced
wrapper that execs a locked-down real binary, or a container image env — with the explicit caveat that a
plain `/etc/profile.d` export is user-removable and therefore not real enforcement.

## Option 2 (PREFERRED): `require_server_password: true` in `/etc/opencode/opencode.json`

**Robustness: STRONG / user-unbypassable.** The managed tier (`/etc/opencode`, root-owned) wins over
`~/.config`/project config (`config.ts:516-522`) and a non-root user cannot edit it. This is the control a
sysadmin actually wants on shared/HPC hosts.

**Effort: MEDIUM — because of the same verified obstacle this whole message is about:** the startup/network
path reads config via `Config.getGlobal()` → `loadGlobal`, which merges **only** `~/.config/opencode/*`
(`config.ts:246-260`) and does **NOT** read the managed `/etc/opencode` tier (that merge is in
`loadInstanceState`, `config.ts:516-534`). So a `require_server_password` key in `/etc/opencode` would be
**silently ignored at startup today.** Three parts:

**(2a) Add the schema key** — `core/src/v1/config/server.ts` (`Server = Schema.Struct({...})` currently only
`port`/`hostname`/`mdns`/`mdnsDomain`/`cors`; unknown keys drop on decode until added):

```diff
 export const Server = Schema.Struct({
   port: Schema.optional(PositiveInt)...,
   hostname: Schema.optional(Schema.String)...,
+  require_server_password: Schema.optional(Schema.Boolean).annotate({
+    description: "Refuse to start the server unless OPENCODE_SERVER_PASSWORD is set to a non-empty value",
+  }),
   ...
 })
```

**(2b) Make the startup path see the managed tier.** Preferred: a small `resolveStartupPolicy()` next to
`resolveNetworkOptions` (`cli/network.ts`) that reads the managed config (reuse
`ConfigManaged.managedConfigDir()` + `readManagedPreferences()`, as `config.ts:516-534` does) and returns
`{ requireServerPassword: boolean }`. Avoids changing `getGlobal` semantics globally (the broader
alternative — merging managed into `loadGlobal` — works but changes global-config behavior everywhere, so
it's riskier). Also read the user/global `server.require_server_password` so the key works from `~/.config`
too, with managed winning.

**(2c) Guard in `Server.listen`** — same seam, reading the resolved policy. Add
`requireServerPassword?: boolean` to `ListenOptions` (`server/server.ts:32-37`); `serve`/`web` pass it from
`resolveStartupPolicy`. OR it with the env flag so both work:

```diff
     const password = process.env.OPENCODE_SERVER_PASSWORD
-    if (Flag.OPENCODE_REQUIRE_SERVER_PASSWORD && (!password || password.trim() === "")) {
+    if ((opts.requireServerPassword || Flag.OPENCODE_REQUIRE_SERVER_PASSWORD) && (!password || password.trim() === "")) {
       return yield* Effect.die(new Error("Refusing to start server: ..."))
     }
```

Sysadmin sets `/etc/opencode/opencode.json` → `{ "server": { "require_server_password": true } }`.
Root-owned, users cannot override it, applies to every listener path. That is the robust answer.

## Comparison

| | Env var (Option 1) | Config key (Option 2) |
|---|---|---|
| Code size | Trivial (~6 lines + 1 flag) | Medium (schema key + startup-policy read + guard) |
| Blocks a shell user from unsecured server | **No** (user can `unset`) | **Yes** (root-owned; user can't edit) |
| Needs new plumbing | No | Yes — startup must read managed config (currently doesn't) |
| Covers `serve`/`web`/TUI in one place | Yes (`Server.listen` seam) | Yes (same seam) |
| Right long-term control | Interim stopgap | **Preferred** |

## Recommendation
- **Preferred: Option 2.** Only it actually prevents a local user from starting an unsecured server, and it
  composes with sibling locks (`server.hostname`, a future `allow_non_loopback_bind: false`) once startup
  reads the managed tier — i.e. it's the same "wire managed config into startup" work this message already
  recommends, plus one key and one guard.
- **Option 1 acceptable as an immediate interim** for service/launcher-only deployments — not enforcement
  on a multi-user host.
- OR both in the single `Server.listen` guard so the interim env var keeps working after the key ships.

## Caveats
- `Effect.die` = fatal error / non-zero exit. For a clean CLI message, do the check in a shared helper
  invoked by `serve`/`web` via their `fail(...)` idiom (`cli/effect-cmd.ts:18`) — but keeping it in
  `Server.listen` is what guarantees no listener path is missed. The seam matters more than the error style.
- Enforcing password *presence* is orthogonal to non-loopback binding and the `/shell`/`/config` issues;
  this only closes the "no password at all" default.
