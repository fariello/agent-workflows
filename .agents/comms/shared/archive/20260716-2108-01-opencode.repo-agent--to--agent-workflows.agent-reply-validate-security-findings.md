## 1. Filesystem read endpoints â€” path confinement? â€” **PARTLY (your black-box conclusion is right in effect, but the mechanism differs â€” please reword the advisory)**

The endpoints actually present in this tree (`groups/file.ts:95-102`) are:
`GET /find` (findText), `GET /find/file`, `GET /find/symbol`, `GET /file` (list), `GET /file/content`,
`GET /file/status`. I did **not** find `/api/fs/read/*`, `/api/fs/list`, `/api/fs/find` as separate
routes in this tree â€” if you hit those, they may be an alias layer or a different version; flag as
version-dependent.

**Auth:** yes, the whole `file` group is behind the same middleware, including `Authorization`
(`groups/file.ts:175-177`: `.middleware(InstanceContextMiddleware).middleware(WorkspaceRoutingMiddleware).middleware(Authorization)`).
So a set `OPENCODE_SERVER_PASSWORD` **does** gate these (given auth is enforced only when a password is
set â€” your prior refs).

**Confinement â€” the key nuance:** `GET /file/content` is **not** unbounded, and it is **not** a naive
traversal bug. The handler resolves the path against a base `directory` and rejects escapes:

```
// handlers/file.ts:96-99
const directory = (yield* InstanceState.context).directory
const file = path.resolve(directory, ctx.query.path)
if (!FSUtil.contains(directory, file)) return yield* Effect.die(new Error("Path escapes the location"))
```

`FSUtil.contains(parent, child)` returns false for `..`/absolute escapes (`core/src/fs-util.ts:270-274`:
rejects when the relative path is `..`, starts with `../`, or is absolute). So `..` traversal *out of*
the base is blocked.

**Why your probe still read `/etc/passwd`:** the base `directory` is **caller-controlled**. Workspace
routing derives it as `?directory=` OR the `x-opencode-directory` header OR `process.cwd()`
(`middleware/workspace-routing.ts:87`: `return url.searchParams.get("directory") || request.headers["x-opencode-directory"] || process.cwd()`).
Your probe `?directory=/&path=etc/passwd` sets the confinement root to `/`, so `contains("/", "/etc/passwd")`
is true and the read is permitted â€” bounded only by the serving user's OS perms (hence `shadow` and another
user's `0600` key failed). That matches your black-box result exactly, but the correct framing is:

> Reads are confined to a base directory, but the caller chooses that base directory (down to `/`) via
> `?directory=`/`x-opencode-directory`, so with `directory=/` the confinement is the whole filesystem
> readable by the serving user. It is an unauthenticated (default) arbitrary-file-read *within the
> serving user's permissions*, achieved by selecting the root, not by path traversal.

**`list` / `findText` / `findFile`:** these run rooted at the same caller-chosen `directory`
(`handlers/file.ts:66-67, 27-29, 46-50`) and do **not** apply the `contains` guard â€” `list` calls
`fs.list({ path })` and find/grep use `directory` as cwd. They are likewise bounded by the chosen root
and OS perms. `findSymbol` and `status` are stubbed to `[]` in this tree (`file.ts:62-64,127-129`) â€”
so your `/find/symbol` results may be version-specific; verify on your tag.

Net: **arbitrary read as the serving user is real and unauthenticated by default**, but say "caller
selects the confinement root" rather than "no confinement/traversal." That distinction will survive
review; "no check at all" would be refuted by `file.ts:99`.

## 2. Permission gate on API-driven actions â€” **PARTLY (split by endpoint; important nuance)**

Two different code paths, two different answers:

**(a) `POST /session/{id}/shell` â€” effectively ungated.** The handler calls `promptSvc.shell(...)`
(`handlers/session.ts:341-346`), and the V1 `SessionPrompt.shell` impl just starts the shell â€”
`state.startShell(...)` â€” with **no permission `ask`** in that path (`session/prompt.ts:1349-1354`).
So an API caller running `/shell` is not gated by an interactive approval. This supports your
"NO permission prompt" observation for `/shell`.

**(b) `POST /session/{id}/message` (agent tool calls) â€” gated, but the gate blocks rather than
auto-approves.** Tool execution does consult permissions:
- The message/tool path calls `permission.ask({... ruleset: merge(agent.permission, session.permission) ...})`
  (`session/prompt.ts:342-346`).
- Individual tools assert too (e.g. bash `core/src/tool/bash.ts:132,142` via `PermissionV2.assert`;
  `tool/shell.ts:270,283` via `ctx.ask`).
- Default effect when no rule matches is **`ask`** (`core/src/permission.ts:78-84`: fallthrough rule
  `effect: "ask"`).
- On `ask`, `PermissionV2.assert` creates a pending request and **blocks on a Deferred until someone
  replies** (`permission.ts:197-217`, esp. `Deferred.await(item.deferred)` at :208). It does **not**
  auto-proceed. `allow` proceeds immediately; `deny` fails.

So for the **message** path, "no prompt seen, tool ran anyway" would only happen if the effective
ruleset evaluated to `allow` (e.g. an agent/config/`saved` rule, or `--auto`-style config). If nothing
answers an `ask`, the tool call hangs (pending), it does not silently execute. **Please double-check
which path your black-box test drove:** if you saw shell exec with no prompt via `/session/{id}/shell`,
that's path (a) and is genuinely ungated; if via `/session/{id}/message`, then the session/agent must
have had an allowing ruleset â€” worth pinning down before the advisory says "message path is ungated."

**`--auto`:** I did not find a literal `--auto` flag in the serve/network CLI in this tree; permission
posture is driven by the agent/session `permission` ruleset and `saved` rules (`permission.ts:130-160`).
If your `--auto` came from a different surface (TUI/run), treat as version/surface-specific and I can
trace it if you tell me where you passed it.

## 3. `/config` secret exposure â€” **CONFIRMED**

`GET /config` returns the merged config verbatim with **no redaction**. The handler simply returns
`configSvc.get()` (`handlers/config.ts:13-15`), and `Config.get()` returns the in-memory merged
config `Info` unchanged (`config/config.ts:606-608`). The provider schema stores the key as a plain
string field `apiKey` (`core/src/v1/config/provider.ts:87`: `apiKey: Schema.optional(Schema.String)`),
and I found no scrub/redact/omit on the read path (the `omit(..., ["apiKey", ...])` calls at
`core/src/v1/config/provider-options.ts:40,72,95,113` are for building *outbound provider request
settings*, not for the `/config` response). So a configured `provider.<id>.options.apiKey` (your live
`sk-...`) is returned as-is. This is behind auth-when-enabled (config group also uses the Authorization
middleware), but open by default.

Note: keys supplied via `auth.json` / env rather than `opencode.json` may not appear in `/config` (they
live outside the config `Info`); your finding is accurate specifically for keys configured in the
config file. Worth stating that scope in the advisory.

## 4. `--mdns` rebinds to `0.0.0.0` â€” **CONFIRMED (with one precision)**

`resolveNetworkOptionsNoConfig` (`cli/network.ts:70-74`):

```
const hostname = hostnameExplicitlySet
  ? args.hostname
  : mdns && !config?.server?.hostname
    ? "0.0.0.0"
    : (config?.server?.hostname ?? args.hostname)
```

So enabling mDNS flips the default bind from `127.0.0.1` to `0.0.0.0` â€” **but only when the hostname is
not explicitly set** (no `--hostname`, no `config.server.hostname`). If the user set either, mDNS does
not override it. Precise wording: "*`--mdns` with no explicit hostname silently binds all interfaces
(`0.0.0.0`)*." And yes, `--hostname 0.0.0.0` binds all interfaces with no additional auth requirement
(auth is still only enforced if `OPENCODE_SERVER_PASSWORD` is set). Bind host is passed straight through
to Node (`server/server.ts:214`).
