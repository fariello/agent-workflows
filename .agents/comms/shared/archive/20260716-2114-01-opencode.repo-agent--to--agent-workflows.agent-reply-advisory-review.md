## 1. Inaccuracies / inconsistencies

Nothing load-bearing is wrong. Items are ordered by importance.

### 1.1 Strengthen the `/shell`-ungated citation (accurate, under-cited)
The claim "`/session/{id}/shell` is genuinely ungated (`session/prompt.ts:1349-1354`)" is **correct**,
but `:1349-1354` is only the thin `shell()` wrapper. The *proof of absence* of a permission gate is in
the `shellImpl` body, `session/prompt.ts:451` through ~`:570`: it writes the user/assistant/tool
records and then spawns directly via `ChildProcess.make(sh, args, { cwd })` → `spawner.spawn(cmd)` with
**no** `permission.ask`/`assert` anywhere in the path (contrast the agent tool path, which does call
`permission.ask`). Recommend citing `session/prompt.ts:451-570 (shellImpl, no permission gate)` alongside
`:1349-1354`. This makes the load-bearing claim reviewer-proof.

Also note the shell runs with `cwd = ctx.directory` (the caller-selected instance directory), i.e. it
is **not** confined to a worktree — worth stating explicitly next to the `/file/content` root-selection
point, since it's the same "caller controls the base" theme.

### 1.2 Version labeling is slightly muddled (cosmetic, but fix for publication)
The header says "Same-user self-test on v1.18.2; cross-user … on v1.18.3 (source refs at commit
`70b56a0a9`)", but §Source references and §134 say the source validation was done at `08fb47373`
(just past v1.18.3). Both commits appear. Recommend one explicit table:
- Installed binary used in self-test: v1.18.2
- Cross-user campaign binary: fresh-install v1.18.3
- Source lines cited: read at `08fb47373` (fork `dev`), to be re-pinned to the matching upstream tag.
`70b56a0a9` is the `v1.18.2` release commit; if the source refs were actually read at `08fb47373`,
drop `70b56a0a9` from the header to avoid implying the line numbers are from v1.18.2.

### 1.3 "no UID / peer-credential check" — true, but add the nuance that peer-cred is *possible*
§30 says there is no UID/peer-credential check. Correct as-is (there is none). For the fix section it's
worth noting Node can obtain peer creds only on a **UNIX socket** (`SO_PEERCRED`); on a loopback TCP
socket there is no portable peer-UID. So fix #3 (peer-cred check on the loopback listener) is weaker than
fix #1 (UNIX socket) and in practice depends on it. See 3.3 below. Not an inaccuracy in the advisory,
but the two fixes are entangled and the advisory lists them as independent.

### 1.4 `/config` gating consistency (confirm — it's fine)
The advisory implies `/config` is open by default but gated when a password is set. Verified: the config
group is behind the same `Authorization` middleware (`groups/config.ts:55-57`), so that's consistent.
No change needed; flagging because it's an easy reviewer question.

### 1.5 `/api/fs/*` route-name caveat is handled well — keep it
The advisory already labels `/api/fs/read|list|find` as version-dependent (not found as separate routes
in the `dev` tree; the confirmed routes are `/find`, `/find/file`, `/find/symbol`, `/file`,
`/file/content`, `/file/status`). That's the honest framing. Just make sure the Impact/Repro sections
that still say "`/find`/`/api/fs` endpoints" (e.g. §73, §111) carry the same "route names are
version-specific; behavior identical" caveat, so a maintainer on `dev` doesn't dismiss the finding when
`/api/fs/*` 404s.

### 1.6 Minor: "`providerID: opencode / modelID: big-pickle`" and "built-in default model"
Fine to keep, but if this goes upstream, note that the presence of a usable default model is what let
T3 run `/shell` without provider config. That's already implied; just don't let a maintainer read it as
"only affects installs with the `opencode` provider."

### 1.7 Terminology consistency
The doc uses "unauthenticated" both for "no password configured (default)" and "reachable without
credentials." Consider one sentence early: "*unauthenticated* here means the default posture where no
`OPENCODE_SERVER_PASSWORD` is set; once set, all cited routes require it." Prevents the misread that the
routes are open even with a password.

None of the above changes the severity or the conclusions.

## 2. Mitigation strategies (operational — deployable today, no code change)

Ordered strongest-first. The advisory's companion hardening doc (`-02`) likely overlaps; this is my
independent list keyed to what the source actually enforces.

### 2.1 Per-user network isolation (strongest; the only real local isolation today)
Loopback is not UID-scoped, so the robust fix is to remove cross-user reachability at the OS layer:
- Run each user's opencode server in a **private network namespace** so its loopback is invisible to
  other users: `systemd --user` unit with `PrivateNetwork=true` (plus `IPAddressDeny=any` for good
  measure), or `unshare -n` wrappers. This defeats `/proc/net/tcp` discovery *and* the connection
  itself, and it survives a mis-set password (defense in depth).
- On HPC, prefer running opencode inside the job's cgroup/namespace (Slurm `--container`/`unshare`) so
  the listener never shares a net namespace with other tenants.

### 2.2 Mandatory password, enforced by the launcher (not the user)
Because the server **fails open** on a missing/mis-set env var (`cli/cmd/serve.ts:15-22`, warning to
stdout only), do not trust humans to export it:
- Wrap `opencode serve`/`web` in a launcher that (a) generates a strong random password, (b) exports
  `OPENCODE_SERVER_PASSWORD` into the child env, (c) refuses to exec if it's unset/empty. Example: a
  `systemd` unit with `Environment=OPENCODE_SERVER_PASSWORD=...` sourced from a `0600`
  `EnvironmentFile=`, or a wrapper script that `exec`s only after setting it.
- Kill stale unsecured servers first (the "stale server still running" gotcha): the launcher should
  `pkill`/`fuser` any existing listener for that user before starting.
- Use the **`Authorization: Basic`** header, never `?auth_token=` (avoids leaking creds into
  `/proc/<pid>/cmdline`, shell history, proxy logs).

### 2.3 Never bind non-loopback on shared/networked hosts
- Forbid `--hostname 0.0.0.0` and `--hostname <routable>`; forbid `--mdns` unless an explicit loopback
  `--hostname 127.0.0.1` is also passed (because `--mdns` silently flips the default bind to `0.0.0.0`
  when hostname is unset — `cli/network.ts:70-74`).
- Enforce via the launcher/wrapper (reject those flags) and, if you run a shared config,
  pin `server.hostname: "127.0.0.1"` in a **managed** `opencode.json` the user cannot override
  (managed-config precedence exists; verify it wins over project config on your version).

### 2.4 Front the port with an authenticating, logging reverse proxy
Since opencode writes **no HTTP access log** (`server.ts:103-104`), if you must expose it:
- Bind opencode to loopback, put nginx/caddy in front on a UNIX socket or authenticated port, and do
  access logging + auth there. This is also your only forensic trail.

### 2.5 Detection / monitoring (compensating control)
- Read-only abuse (`/file/content`, `/session`, `/config`) leaves no opencode-side trace; only
  session-creating/`/shell` actions land in `~/.local/share/opencode/opencode.db`. So:
  - Monitor for unexpected **listeners** per user (`ss -tlnp` sweep; alert on non-loopback binds).
  - Monitor the DB for sessions whose `directory` or activity doesn't match the owning user's expected
    projects (stealth attacker sessions persist there).
  - Prefer network-layer logging (proxy/eBPF/`ss` sampling) since the app won't tell you.

### 2.6 Policy
- Treat "runs a listening opencode server" as the gating precondition (the advisory is right that a
  plain TUI opens no port). Restrict *who* may run `serve`/`web` on shared hosts; single-user boxes are
  unaffected.

## 3. Possible code fixes (upstream) — concrete, mapped to source

The advisory's fix list (§104-114) is good. Below I make each **actionable against the actual code**,
add priority, and add three the advisory under-specifies (timing-safe compare, managed-config bind lock,
default-ruleset audit).

### 3.1 [HIGH] Fail-closed auth option + generated token (advisory #2, #7-fail-open)
- Add a config/flag `server.requireAuth: true` (and/or `--require-auth`) that makes `serve`/`web`
  **refuse to start** when no password is set, instead of the stdout warning at `cli/cmd/serve.ts:15-22`.
- Better default: if no password and stdin is non-interactive/service mode, **generate** a random
  per-instance token, write it `0600` to `Global.Path.state/server-auth` (or print once), and require
  it. `ServerAuth.required()`/`authorized()` (`server/auth.ts:24-33`) already centralize the check;
  wire the generated token into `ServerAuth.Config`.
- Smallest viable change: a boolean that flips the missing-password branch from warn→exit.

### 3.2 [HIGH] UNIX-domain-socket listener with `0700` dir / `0600` socket (advisory #1)
- `server/server.ts:200,214` builds a Node TCP server (`createServer()` + `NodeHttpServer.layer({ port,
  host })`). Node's `http` server can `listen({ path })` on a UNIX socket. Add a `--socket <path>` (and
  `server.socket` config) that, when set, listens on a UDS created in a `0700` dir, and skips the TCP
  bind. This gives OS-enforced per-user isolation — the only robust local boundary.
- Update the SDK/CLI client + `attach`/`run --attach` to accept a socket path.

### 3.3 [MED] Peer-credential (UID) check — only meaningful on the UDS (advisory #3)
- On a UNIX socket, add an optional `SO_PEERCRED` check rejecting connections whose peer UID != server
  UID. On loopback TCP there is no portable peer UID, so this fix is effectively *paired with* 3.2, not
  independent. Recommend the advisory merge #1 and #3 into "UDS + peer-UID."

### 3.4 [HIGH] Redact secrets from `/config` (advisory #4)
- `handlers/config.ts:13-15` returns `configSvc.get()` (`config/config.ts:606-608`) verbatim; the
  provider `apiKey` is a plain string (`v1/config/provider.ts:87`). Add a redaction pass on the response
  that replaces `provider.*.options.apiKey` (and any `*.options.{apiKey,authToken,token,headers.
  authorization}`) with a sentinel (`"***redacted***"`), or omit them. Keep the raw value only for the
  internal consumers that need it — the HTTP boundary should never emit it.
- Note scope: keys in `auth.json`/env don't flow through `/config` anyway; this is about config-file
  keys.

### 3.5 [HIGH] Gate `/shell` behind the permission policy (advisory #5)
- `shellImpl` (`session/prompt.ts:451-570`) spawns the command with no `permission.ask`. Route it
  through the same `permission.assert({ action: "shell"/"bash", resource: command, ... })` the agent
  tool path uses (`permission.ts:197-217`), so the default `ask` effect blocks pending approval and a
  configured `deny` refuses. This closes the load-bearing exec primitive even when auth is off (defense
  in depth) and makes API-driven exec consistent with interactive turns.
- Complementary: audit **why the default session ruleset evaluated to `allow`** in the self-test
  message path (advisory §41 leaves this open). If a fresh/default agent resolves to an allowing
  ruleset for filesystem-mutating actions, that's a second issue. Trace `agents.resolve(...).permissions`
  vs `missingAgentPermissions` (`permission.ts:15,143-146`) for the default agent.

### 3.6 [MED] Confine filesystem endpoints to the worktree, not caller-selected root (advisory #6)
- The `contains` guard (`handlers/file.ts:96-99`, `fs-util.ts:270-274`) is correct but rooted at the
  caller-chosen `directory` from `workspace-routing.ts:87` (`?directory=`/header/`cwd`). Options:
  (a) require the resolved path to be within the *session/project worktree* rather than an arbitrary
  caller-supplied base; (b) reject `directory` values that aren't the instance's known project dir; or
  (c) at minimum, refuse `directory=/` and paths outside the worktree. Apply the same base to
  `list`/`find`/`grep`, which currently skip the `contains` check.
- This alone converts "arbitrary read as the serving user" into "read within the project," which is the
  intended surface.

### 3.7 [MED] Optional HTTP access log (advisory #7)
- `server/server.ts:103-104` sets `disableLogger: true`, `disableListenLog: true`. Add a
  `--access-log` / `server.accessLog` that re-enables per-request logging (method, path, status, remote,
  auth-subject) to `Global.Path.log`. Makes abuse auditable from opencode itself.

### 3.8 [MED] Refuse/loudly-gate non-loopback bind without auth (advisory #8)
- In `resolveNetworkOptions`/`serve` (`cli/network.ts:70-74`, `cli/cmd/serve.ts`), if the resolved
  `hostname` is non-loopback (incl. the `--mdns`→`0.0.0.0` path) and no password is set, **refuse to
  start** (or require `--i-know-this-is-exposed`). At minimum escalate the warning and send it to stderr
  (currently stdout `console.log`).

### 3.9 [LOW] Timing-safe password comparison (not in the advisory — add it)
- `authorized()` compares with `===` (`server/auth.ts:32`), which is not constant-time. For a
  network-exposed (`0.0.0.0`) server this is a (minor) side channel. Use a constant-time compare
  (`crypto.timingSafeEqual` on equal-length buffers). Low severity but trivial and expected in a
  security fix PR.

### Suggested fix priority for the disclosure
P0: 3.4 (`/config` redaction), 3.5 (`/shell` gate), 3.1 (fail-closed). These are the exec + secret
primitives and the fail-open default.
P1: 3.2/3.3 (UDS + peer-UID), 3.6 (fs confinement), 3.8 (non-loopback guard).
P2: 3.7 (access log), 3.9 (timing-safe compare).

## 4. Bottom line

  label to tidy (`70b56a0a9` vs `08fb47373`), and keep the version-dependent-route caveat consistent

  timing-safe compare and a default-ruleset audit, and I'd merge the UDS + peer-UID items since
