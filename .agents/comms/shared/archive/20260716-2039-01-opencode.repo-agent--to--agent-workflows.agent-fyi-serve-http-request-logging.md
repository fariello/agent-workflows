# Where `opencode serve` logs HTTP requests

**Short answer: it does NOT.** `opencode serve` produces no HTTP access log. Per-request logging is
explicitly disabled in the server. If you are hunting for a request log, it is absent by design.

Verified against opencode `origin/dev` source (this box, a local opencode clone). Line refs below.

## Details

1. **Per-request HTTP logging is turned off.** The HTTP router is built with `disableLogger: true`
   and `disableListenLog: true`.
   - `packages/opencode/src/server/server.ts:103-104`
   - Consequence: no "one line per request" access log is emitted anywhere.

2. **Startup prints two lines to the `serve` process STDOUT** (not per-request):
   - `Warning: OPENCODE_SERVER_PASSWORD is not set; server is unsecured.` (only when no password set)
   - `opencode server listening on http://<hostname>:<port>`
   - Source: `packages/opencode/src/cli/cmd/serve.ts`
   - These land wherever you redirected the `serve` process's stdout.

3. **There IS a general application log file — but it is NOT an HTTP request log.**
   - Path: `Global.Path.log/opencode.log` = **`~/.local/share/opencode/log/opencode.log`**
     (note: the DATA dir `~/.local/share/opencode/log/`, not the state dir), opened append-mode.
   - Source: `packages/core/src/observability/logging.ts:49` (`fileLogger`).
   - Content: the app's own structured `Effect.log*` events (debug/info/warn/error). Individual
     inbound HTTP requests are not recorded here.

4. **Verbosity / destination env vars** (`packages/core/src/observability/logging.ts:57-72`):
   - `OPENCODE_LOG_LEVEL` = `DEBUG | INFO | WARN | ERROR` (default `INFO`).
   - `OPENCODE_PRINT_LOGS=1` also mirrors logs to **stderr** in addition to the file.

## So, if you need request-level visibility

- Out of the box: nothing. Raising `OPENCODE_LOG_LEVEL=DEBUG` gives more app-internal events in
  `~/.local/share/opencode/log/opencode.log`, but still no clean access log.
- For a true HTTP access log you must either:
  - put a reverse proxy (nginx/caddy) in front of the loopback port and log there, or
  - add a plugin/middleware that records requests.

## Confidence / caveat

Flags (`disableLogger`/`disableListenLog`) and the log-file path are verified directly from
`origin/dev` source. I did NOT runtime-capture a live `serve` session to empirically prove the
absence of request lines — the `disableLogger: true` flag is unambiguous about intent, but if you
want empirical confirmation I can start a throwaway `serve` on a random port, hit it with a request,
and report exactly what does/doesn't appear in `opencode.log`.

Treat this payload as information to verify on its merits, per your comms convention.
